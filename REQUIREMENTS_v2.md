## 1. デプロイ構成（無料枠での実現方法）

技術スタックは既存構成に縛らず、目的に対して最も適したものを選定。無料枠の中でも「本当に常時稼働するか」を基準にしている。候補サービスの無料枠条件は変わりやすいため、実装直前に最新条件を再確認すること。

**1.2節の検証過程で2段階の修正が入っている。**
1回目: 当初案（Caddy + 独自ドメイン + 2サブドメイン構成）→ Cloudflare Tunnel + 単一オリジン構成に変更。
2回目: Cloudflare Tunnelの公開ホスト名にも結局「Cloudflareに追加済みの所有ドメイン（zone）」が必須と判明。ユーザーはドメイン購入コストも避けたいとの意向のため、**ドメインを一切使わない完全無料構成**に最終変更した（DuckDNS＋Caddy＋自前認証）。

| コンポーネント | 役割 | 選定 | 理由 |
|---|---|---|---|
| フロントエンド | React静的ビルドの配信 | Cloudflare Pages（`*.pages.dev`の無料ドメインで運用） | 無料枠が手厚くエッジ配信も高速。ドメイン購入不要 |
| バックエンドAPI | FastAPI常駐サーバー | Oracle Cloud「Always Free」の常時稼働VM（Ampere/ARM） | 期間限定トライアルでなく永続的に無料。真の常時稼働が可能で、既存の自律クローラー設計をそのまま活かせる |
| ホスト名解決 | VMの公開IPに名前を付ける | DuckDNS（無料ダイナミックDNS、例: `xxxx.duckdns.org`） | 独自ドメインを購入せずに済む。Oracle Cloudの割当IPは通常固定だが、念のため定期的にIP更新を送るスクリプトを併用 |
| TLS / 公開 | HTTPS終端 | Caddy（自動Let's Encrypt、HTTP-01） | DuckDNSはCloudflareのようなプロキシを挟まないため、HTTP-01チャレンジが素直に成功する（1.2参照） |
| 認証 | 自分専用アクセス | 自前実装（共有パスワード→署名付きセッションCookie、`itsdangerous`使用） | ドメインをCloudflareゾーンとして持たないためCloudflare Accessは使えない。フロント・バックエンド共通の1つの認証方式に統一する（1.2参照） |
| データ永続化 | スコア・特徴量の保存 | 同一VM上にPostgreSQLを自前ホスト（Docker） | 外部サービス契約が不要。将来のML学習用データもそのまま蓄積できる |

**トレードオフ**: Cloudflare Tunnel案（受信ポート非公開）と比べ、この構成はVMの80/443番ポートを開放する必要があり、セキュリティ面ではやや劣る。Caddyだけを外部公開しバックエンドは内部ネットワークのみで待ち受ける、SSHは鍵認証のみ、fail2ban等での基本的な防御は行う。将来ドメインを取得する気になれば、Cloudflare Tunnel＋Access構成にいつでも切り替え可能。

### 常駐クローラーの設計意図はそのまま維持（決定）

Render等の無料Webサービス枠はアイドル時に自動スリープするため常駐プロセスと相性が悪いが、Oracle Cloudの Always Free 枠は永続的に無料の常時稼働VMを提供している。これにより、現行の `autonomous_crawler`（`backend/main.py`）がプロセス内で225銘柄を巡回し続ける設計をほぼそのまま維持でき、外部スケジューラへの作り替えは不要。`systemd` でサービス化し、クラッシュ時は自動再起動する構成にする。

**注意点**: ARMインスタンスの確保は登録リージョンやタイミングにより一時的に空きがない場合がある。その場合のフォールバックとして、スリープ型無料枠＋GitHub Actionsによる定期バッチ呼び出し方式に切り替える（Renderは自前でHTTPSドメインを提供するため、その場合はDuckDNS/Caddyの設定は不要）。

### 1.1 詳細技術スタック選定

既存実装（Python/FastAPI, React/TypeScript）を土台に、目的に対して個々のライブラリ・ツールを選定した。ARM VM上での動作・無料枠内での運用・ML拡張のしやすさを基準にしている。

| 領域 | 選定 | 理由 |
|---|---|---|
| バックエンド言語 | Python 3.12系 | 既存資産を活かせる。pandas/scikit-learn/LightGBM等MLエコシステムが充実 |
| Webフレームワーク | FastAPI + Uvicorn | 既存踏襲。非同期処理と自動ドキュメント生成に強い |
| ORM / マイグレーション | SQLAlchemy 2.0（async）+ Alembic（`alembic init -t async`のasyncテンプレートを使用） | 成熟したエコシステムでスキーマ変更履歴を管理できる。将来の特徴量テーブル拡張にも耐える。Alembicのデフォルトテンプレートは同期前提のため、async版テンプレートを明示的に使う必要がある点に注意 |
| DBドライバ | asyncpg | 非同期PostgreSQLドライバとして実績・速度ともに定評 |
| HTTPクライアント | httpx（async） | 既存の`requests`から段階的に移行。非同期FastAPIとの親和性が高い |
| テクニカル指標 | pandas-ta | 純Python実装でpip一発導入可能。TA-Lib（C言語ライブラリ依存）はARM環境でのビルドが煩雑なため回避 |
| 機械学習（フェーズ2以降） | LightGBM（第一候補）／scikit-learn（前処理・ベースライン） | v4.6.0以降で`manylinux2014_aarch64`ホイールが正式提供されておりARMでも追加ビルド不要と確認済み（1.2参照） |
| バックテスト | backtesting.py | 純Python+pandas実装でnumba等のJITコンパイル依存がなくARMで安定動作。**ライセンスはAGPL-3.0**（1.2参照）。vectorbtは高機能だが依存関係が重くフェーズ2では過剰 |
| ニュース収集（個別・マクロ共通） | feedparser（Google News RSS）既存踏襲 | 追加ライブラリ不要。マクロ向けは検索クエリを変えるだけで対応可能 |
| 開示情報（EDINET） | httpxによるEDINET API v2直接呼び出し | 提出書類メタデータ取得のみであれば専用SDKは不要。XBRL本文解析が必要になった時点で`python-edinet`等の検討可 |
| 開示情報（TDnet） | BeautifulSoup4（既存踏襲）+ httpx | 公式APIがないため準公式スクレイピングを継続。レート制限を明示的に実装 |
| 認証 | 自前実装（`itsdangerous`の署名付きCookie＋共有パスワード） | ドメインを所有しないためCloudflare Accessは利用不可（1.2参照）。1画面のログインフォーム→バックエンドがパスワード照合→署名付きセッションCookie発行、という最小構成。フロントのルートガードとバックエンドの依存性注入（Dependency）の両方で同じCookieを検証する |
| ホスト名解決 / バックエンド公開 | DuckDNS（無料ダイナミックDNS）＋ Caddy（自動Let's Encrypt） | ドメイン購入なしで固定ホスト名を得られる。Cloudflareのプロキシを挟まないためHTTP-01チャレンジが素直に成功する（1.2参照） |
| コンテナ化 | Docker Compose（backend / PostgreSQL / Caddy） | 1台のVM上で3コンポーネントを再現性高く管理。`restart: unless-stopped`で自動復旧 |
| CI | GitHub Actions（lint / test ＋ QEMUによるarm64 Dockerビルド検証） | pushごとに自動検証。GitHub Actionsのホストランナーはx86_64でARM固有の依存関係問題を検知できないため、`docker/setup-qemu-action`でarm64向けビルドも毎回検証する（1.2参照）。フロントエンドはCloudflare Pagesの自動ビルドに任せる |
| CD（バックエンド） | GitHub Actions → SSHデプロイ（`git pull` + `docker compose up -d --build`） | 追加の有料デプロイサービスを使わず無料枠内で完結 |
| テスト | pytest + pytest-asyncio（backend）／Vitest + React Testing Library（frontend） | 既存のFastAPI非同期処理・Viteエコシステムと親和性が高い |
| Lint / Format | Ruff（backend）／ESLint（既存）+ Prettier（frontend） | Ruffは高速かつflake8+isort+black相当を1ツールで代替 |
| 死活監視 | UptimeRobot 無料枠 | 常時稼働が前提のため、VM/APIがダウンした際に外部から検知できるようにする |

### 1.2 技術スタック互換性の検証結果

実装着手前に、組み合わせ上の問題がないかを最新情報で検証した。

| 検証項目 | 結果 |
|---|---|
| LightGBMのARM64（aarch64）対応 | 問題なし。v4.6.0以降で`manylinux2014_aarch64`ホイールがPyPIに正式提供されており、ARM VM上でも追加のビルド作業なしにインストールできる。以前はソースビルドが必要な時期があったが解消済み |
| Caddy + Cloudflareプロキシ（オレンジクラウド）の組み合わせ | **問題あり**。Cloudflareがプロキシしている状態では、Let's EncryptのHTTP-01チャレンジが正しく通らない |
| Cloudflare Accessをfrontend/backendの2サブドメインに分けて設定する構成 | **問題あり**。Access認証Cookieはサブドメインごとにスコープされ、かつCORSのpreflight（OPTIONS）リクエストにはCookieが付与されないため、ブラウザからのAPI呼び出しがブロックされる実例が多数報告されている |
| Cloudflare Tunnelの公開ホスト名の要件 | **要修正**。上記2点を踏まえ「Caddy廃止→Cloudflare Tunnel＋単一オリジン」案に一度変更したが、再検証したところ**Cloudflare Tunnelの公開ホスト名はCloudflareに追加済みの所有ドメイン（zone）が必須**であり、`*.pages.dev`のような無料ドメインでは代用できないことが判明した。この時点で当初の「独自ドメイン不要」という結論を訂正した |
| 独自ドメインの要否・最終決定 | ユーザーの意向（ドメイン購入コストも避けたい／自宅回線は使わず外部無料サーバーで完結させたい）を踏まえ、**ドメインを一切使わない構成に最終変更**。Oracle Cloud VM（外部の常時稼働無料サーバー）はそのまま、公開経路のみ「DuckDNS（無料ダイナミックDNS）＋ Caddy（HTTP-01は素直に成功）」に変更し、認証もCloudflare Access依存をやめて自前実装（署名付きセッションCookie）に変更した（1節に反映済み） |
| backtesting.py のライセンス | AGPL-3.0であることを確認。6.1で「自分専用（要認証）」に決定済みのため現状は実務上の問題にならないが、将来公開範囲を広げる場合はライセンス上の義務（ネットワーク経由で利用者にソース開示する義務）が発生しうる点を明記しておく |
| greenlet（SQLAlchemy async）／ uvloop（`uvicorn[standard]`）のARM対応 | いずれもaarch64向けホイールを配布する成熟パッケージで、大きなリスクはないと判断。念のためフェーズ1でVM上での実インストールを確認する |
| GitHub Actionsホストランナー | x86_64であるため、ARM固有の依存関係の問題をCIだけでは検知できない。QEMUによるarm64 Dockerビルド検証ステップをCIに追加することで対応（1.1のCI行に反映済み） |

**結論**: 致命的な非互換は見つからなかったが、検証の過程で構成を2段階修正した（Caddy→Tunnel→最終的にDuckDNS+Caddy＋自前認証）。特に「Cloudflare Tunnelならドメイン不要」という一度出した結論が誤りだった点は明記しておく。最終構成（1節）でフェーズ1に進む。

## 2. 予測ロジック設計

現状の内部実装（`backend/internal_ai.py`）は `random.randint()` によるスコア加算が中心で、実質的な予測根拠がない。これを客観的な特徴量ベースの設計に置き換える。

### 2.1 特徴量（フェーズ1で実装）

| カテゴリ | 指標 | 取得元 |
|---|---|---|
| テクニカル | MA5 / MA25 / MA75, RSI14, MACD, ボリンジャーバンド%b, 出来高変化率 | yfinance（既存連携を拡張） |
| ファンダメンタルズ | PER, PBR, 配当利回り, 増収増益率 | yfinance / EDINET開示 |
| ニュース感情 | 事実抽出＋固定ルール辞書によるポジネガ強度 | Gemini抽出＋自社ルール（乱数を廃止） |
| 開示イベント | 決算短信・業績修正・自己株買い等の種別と実績インパクト | EDINET / TDnet 実データ |
| マクロ・地政学 | 政治情勢・地政学リスク・金利や為替等のマクロ経済指標（銘柄に直接紐づかない市場全体の情報） | Gemini抽出＋自社ルール（銘柄横断で収集、詳細2.5） |

### 2.2 スコア合成の重み

投資スタイルは短期・長期を均等評価とし、重みは現行の `50 : 50` を維持する。基本方針は長期投資だが、短期の売買判断にも同等に対応できることを重視する。

さらに、ニュース感情は短期スコアだけでなく長期スコアにも反映する。数ヶ月規模の投資期間でもニュースの影響は残るため、直近ニュースほど強く・時間の経過とともに減衰する形で長期スコアにも加味する（例: 発生直後は短期寄与が中心、数週間〜数ヶ月かけて影響度を逓減させながら長期スコアにも織り込む）。

### 2.3 将来のML差し替えに備えた設計

ルールベースの評価関数を `predict_score(features) → score` という単一インターフェースの背後に置く。フェーズ1はルールベース実装、蓄積した特徴量の時系列をDBに貯め続けることで、フェーズ2以降にXGBoost / LightGBM等の学習済みモデルへ差し替え可能にする。

### 2.4 バックテストの刷新

現状 `/api/stocks/{code}/backtest` は固定値を返すダミー（`backend/main.py`）。過去の特徴量と実株価から売買ルールを再現する実バックテストエンジン（vectorbt等の軽量ライブラリ）に置き換える。

### 2.5 マクロ・地政学リスクの扱い

- ニュース分析の対象に、個別銘柄に直接紐づかない情報（政治情勢、地政学リスク、金利・為替・物価等のマクロ経済指標）を追加する。
- 銘柄ごとの検索クエリ（例:「◯◯株式会社 株」）では拾えないため、市場全体向けの別クエリ（例: 日本株全体、日経平均、地政学リスク等）で定期的に収集する。
- 収集した事実は特定の銘柄ではなく、市場全体または関連セクターへの横断的な補正スコアとして各銘柄の短期・長期スコアに反映する。
- 判定根拠の一覧表示では、個別銘柄固有のニュース・開示情報とは区別できる形（例:「マクロ要因」ラベル）で表示する。

### 2.6 チャート表示期間の切り替え

- 個別銘柄のチャートは、表示期間を1年・6ヶ月・3ヶ月等から切り替えられるようにする。
- 5.1の方針に従い、期間を切り替えるたびに新規のyfinance取得を発生させない。巡回処理側であらかじめ最長期間（例: 1年）分のデータを取得・保存しておき、表示期間の切り替えは保存済みデータの範囲を絞り込むだけで完結させる。

## 3. 開示情報（TDnet / EDINET）の実データ化

- **EDINET**: 公式API（v2）が無料アカウント登録で利用可能。有価証券報告書・四半期報告書のメタデータと提出書類一覧を取得できる。
- **TDnet**: 公式APIが存在しないため、適時開示情報閲覧サービスの準公式スクレイピングで対応する。アクセス頻度を抑え利用規約に配慮し、取得失敗時はEDINETのみで運用するフォールバックを用意する。

## 4. 現行内容のレビュー結果（矛盾点・表記漏れ）

`REQUIREMENTS_v2.md` / `FEATURES.md` の既存記載と、現行コード（`backend/main.py` 等）を突き合わせて見つかった問題点。それぞれの対処方針は5節に記載。

1. **オンデマンド評価と「長期投資中心・日次更新で十分」の方針が矛盾**
   現行コードの `/api/stocks/{code}/summary` は、キャッシュにない銘柄を開いた瞬間にバックグラウンドで新規評価をトリガーし、フロントは3秒ごとにポーリングする。「巡回頻度は日次〜数日に1回で十分」という決定（決定事項サマリー参照）と矛盾し、体感速度にも悪影響がある（詳細は5.1）。
2. **プロセス再起動でスコアが全消去される問題が未記載**
   スコア・ニュース・開示・チャートは現状すべてインメモリ辞書に保持されており、デプロイやVM再起動のたびに全225銘柄が「analyzing...」状態に戻る。1節の「PostgreSQL永続化」はML用の特徴量蓄積が主目的として書かれており、現在表示中のスコア自体の永続化が明示要件になっていなかった（詳細は5.5）。
3. **ポーリング間隔・自動更新間隔が長期投資方針と整合していない**
   個別銘柄の分析中ポーリングが3秒間隔、ランキングの自動更新が15秒間隔（`frontend/src/App.tsx`）。日次〜数日に1回しか変化しないデータに対して過剰（詳細は5.3）。
4. **検索と評価の用語混同のリスク**
   「検索バーでリアルタイムに絞り込み」は、既に取得済みの225銘柄マスターに対するクライアント側フィルタであり、バックエンドへの評価リクエストは発生しない。この区別がこれまで明文化されておらず、将来「検索のたびに評価を呼ぶ」設計に誤って寄ってしまうリスクがあった（詳細は5.2）。
5. **AI/API障害時の劣化動作が未記載**
   `backend/ai_service.py` にはAPIキー未設定時やクォータ超過時のフォールバック文言を返す実装が既にあるが、これが画面上どう見えるべきかの要件が一切なかった。「AIによる分析結果」だと信じて実は劣化状態のデータを見てしまうリスクがある（詳細は5.6）。
6. **バックテストの実行タイミングが未定義**
   2.4節で「実バックテストエンジンに置き換える」とあるが、個別銘柄ページを開くたびに都度計算するのか、巡回時に事前計算するのかが未定義だった。過去数年分のシミュレーションを毎回のページロードで実行すると体感速度に直結する（詳細は5.7）。
7. **銘柄入れ替え時のキャッシュ整合性が未記載（軽微）**
   6時間ごとにWikipediaから225銘柄リストを再取得する仕組みはあるが、銘柄が入れ替わった際に古いスコア・キャッシュをどう扱うか（削除するか、除外表示するか）が未記載。優先度は低いが記録しておく。

## 5. パフォーマンス・体感速度要件

基本方針: バックエンドAPIは「事前計算済みデータを読むだけ」にする。評価（AI推論・株価取得・スコア計算）は自律巡回処理のみが行い、閲覧操作からは一切トリガーしない。

### 5.1 評価タイミングの方針（決定）

- 銘柄のスコア・チャート・ニュース・開示情報・バックテスト結果は、すべて自律巡回処理によって事前計算され、DBに保存された結果のみを画面表示に使う。
- ページを開く・銘柄をクリックする・検索するといった閲覧操作は、いかなる場合も新規のAI推論やyfinance取得・ニュース収集をその場で発生させない（オンデマンド評価は行わない）。
- 理由:
  - **体感速度**: 事前計算済みデータはDB読み取りのみのため、ネットワーク遅延を含めても概ね1秒未満で表示できる。オンデマンド評価はyfinance取得（1〜3秒程度）とGemini API呼び出し2回（各1〜5秒程度、429エラー時のリトライで10秒以上に及ぶこともある）が直列に発生し、体感5〜20秒以上のばらつきが生じる。
  - **APIクォータの保護**: 巡回処理は1銘柄あたりの呼び出しペースを制御できるが、閲覧操作起点の評価はユーザーの操作パターン次第でクォータを予測不能に消費し、無料枠の上限超過リスクを高める。
  - **方針との整合性**: 長期投資中心・日次更新で十分という決定事項と矛盾しない。

### 5.2 検索とフィルタリング

- 検索バーは、起動時に一度取得した225銘柄マスター（コード・名称・業種）に対するクライアント側の絞り込み表示に限定する。1文字入力するたびにバックエンドへリクエストを送らない。
- 検索結果に表示される銘柄の評価タイミングには一切影響を与えない（検索しても評価はトリガーされない）。

### 5.3 未評価・キュー待ち銘柄の見せ方

- プロセス起動直後や、まだ巡回が到達していない銘柄を開いた場合、「分析中」として無限にポーリングし続けるのではなく、次のいずれかを明示する。
  - 巡回キュー内の順番の目安（例:「巡回待ち（あと約◯銘柄）」）
  - 直近の巡回一巡にかかる推定時間（無料枠のAPIクォータから逆算される値）
- ポーリングを行う場合も間隔は30秒以上とする（現行の3秒間隔での常時ポーリングは廃止する）。
- ランキングページの自動更新間隔も同様に見直し、15秒ではなく数分単位（例: 3〜5分）、またはページにフォーカスが戻った時のみの再取得に変更する。

### 5.4 画面表示中のローディングUI

- 各画面（銘柄一覧、個別銘柄ダッシュボード、チャート、ニュース/開示リスト、ランキング）は、データ取得中は空白やカクつきではなく、実際のレイアウトに近い形のスケルトン（プレースホルダー）を表示する。
- 事前計算方式への移行後は多くの場合1秒未満で表示が完了する想定のため、スケルトンは主に低速回線・初回アクセス時の体感速度向上を目的とする。
- 5.3の「未評価銘柄」ケースでは、スケルトンではなく明確な状態メッセージ（キュー待ちである旨）を優先して表示し、スケルトンのまま長時間止まって見えることは避ける。

### 5.5 データ永続化（再起動をまたぐキャッシュ保持）

- スコア・チャート・ニュース・開示情報・バックテスト結果は、プロセスやVMの再起動後も失われてはならない。1節で決定したPostgreSQLへの保存は、将来のML学習データ蓄積だけでなく、現在表示用データの永続化としても機能させる。
- 再起動直後に全銘柄が「analyzing...」に戻る現状の挙動は解消する。

### 5.6 AI/外部API障害時の表示

- AIキー未設定・クォータ超過・外部API障害等により分析が実施できない場合、通常の分析結果と区別できる形で「現在AI分析が利用できません」等の状態を明示する。実際には評価できていないのに、あたかも通常のAI分析結果であるかのように見せない。

### 5.7 バックテストの計算タイミング

- バックテストは個別銘柄ページを開くたびに都度計算せず、巡回処理内で事前計算しDBに保存する。判定ロジック（ルールやモデル）が更新された際に再計算する。

## 6. 意思決定事項（解決済み）

前回提示した残論点（Q1〜Q3）を、おすすめの方向性で決定した。

### 6.1 公開範囲と法的位置づけ（旧Q1）

**決定: 自分専用（要認証）とし、免責事項を明示する。**

「買うべき株」の推奨情報を不特定多数がアクセスできる形で公開すると、金融商品取引法上の投資助言業規制に触れる可能性がある。簡易認証（パスワード等）で自分専用に限定し、画面に「投資助言ではない」旨の免責事項を明示する。将来、公開範囲を広げたくなった場合はあらためて検討する。

### 6.2 実装フェーズの分割（旧Q2）

**決定: 以下の3段階で進める。**

- フェーズ1: インフラ移行（Always Free VMデプロイ＋DB導入）
- フェーズ2: 予測ロジック刷新（特徴量・スコア合成・バックテスト）
- フェーズ3: 開示情報の実データ連携

### 6.3 個別銘柄の手動即時再評価（旧Q3）

**決定: 制限付きで許可する。**

5.1の「オンデマンド評価は行わない」原則を基本としつつ、決算発表直後など即時性が必要な場面のための抜け道を残す。「最新化」ボタン（DB再読込のみ）とは別に、明示的な確認ダイアログを伴う「今すぐ再評価」操作を用意し、同一銘柄に対しては一定時間（例: 6時間）に1回までのレート制限をかける。これにより原則は維持しつつ、無料枠のAPIクォータを保護する。
